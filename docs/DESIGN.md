# Design: Interactive STEM Visualizer — 3 Mental Models Per Concept

## Problem Statement

Students learning STEM hit concepts and problems that don't click from static
text. They want to *understand* (build intuition), not just get an answer. The
proposed product takes a STEM concept, formula, or problem as input and produces
3 interactive visuals that let the student manipulate variables and see the idea
from multiple angles, so it sticks.

Target market: high school and college STEM students.

## Demand Evidence

**None yet.** This is honest and important. The founder confirmed there is no
paying user, no waitlist, no "when does it ship" pressure, and no specific person
who would be upset if it vanished. The hunch is that visual/interactive learning
is underserved. Everything downstream of this doc is therefore a hypothesis, and
the assignment is a cheap validation test, not a build sprint.

This matters because "interactive visuals help people learn" is the exact
sentence that has launched and killed dozens of ed-tech tools. It sounds
obviously true, which is why it's dangerous: people say "that's cool" for free,
then go back to YouTube and ChatGPT during exam week.

## Status Quo

When a STEM concept won't click, students today reach for (in order of frequency,
per founder):

1. **YouTube / Khan Academy** — watch someone explain it visually. Free, trusted,
   already visual, enormous library. The hardest incumbent. Weakness: generic —
   it explains *a* concept, not *the* specific problem on the student's worksheet.
2. **ChatGPT / AI chat** — ask for a step-by-step explanation. Fast-growing.
   Weakness: historically a wall of text. **But as of 2026, OpenAI shipped
   interactive adjustable visual modules inside ChatGPT for 70+ core math/science
   concepts aimed at exactly this audience** (see Premises + Open Questions). This
   is the single most important competitive fact in this session.

The seam between them: neither gives a visual that is BOTH generated for the exact
thing in front of the student AND manipulable AND offers multiple representations.
YouTube is visual-but-generic; ChatGPT's new feature is interactive-but-
concept-triggered (one of 70 curated topics, one module).

## Target User & Narrowest Wedge

**Market:** HS + college STEM students (founder's explicit call — wide market).

**Validation wedge (revised P2):** The founder chose to keep the product BROAD
("paste anything") rather than narrow it to one course. Reconciliation reached in
session: keep the input broad in the product, but treat breadth as a *discovery
instrument* during validation. Instrument heavily and let one cohort/topic emerge
as the beachhead from real repeat-usage data, rather than declaring a beachhead up
front or declaring victory on scattered one-time reactions.

**Smallest paid-worthy version:** A web app where a student pastes a concept or a
specific problem and gets 3 interactive, *correct* visuals — three different
mental models of the same idea across distinct representation categories (e.g., a
manipulable graph, a physical analogy, and a step-by-step interactive walkthrough). The
"3 = 3 representations" framing is the actual thesis, not a quantity; show fewer than 3
when fewer valid distinct representations exist rather than padding.

## Constraints

- Pre-product, solo founder, no users, no revenue, no repo yet.
- Correctness is non-negotiable for a *learning* tool: a confidently wrong
  interactive visual is worse than none and instantly destroys trust.
- Must differentiate from ChatGPT's now-shipping interactive STEM visuals.
- Web app (no app-store distribution friction).

## Premises

Confirmed with the founder (P2 was revised after pushback; all others agreed):

- **P1 (agreed):** Generic "concept → interactive visual" is being commoditized by
  ChatGPT's new feature. The defensible wedge must lean on what a concept-triggered,
  single-module, top-70-only feature structurally can't do: the *specific* problem
  (the 71st concept / the actual worksheet), multiple simultaneous representations,
  and study persistence.
- **P2 (REVISED after founder pushback):** Original premise was "narrow to one
  beachhead course." Founder disagreed and chose a broad product. Revised premise:
  build broad, but instrument validation so breadth discovers the beachhead. The
  success metric is repeat usage concentrated in some subject/topic, not breadth of
  one-time "neat!" reactions.
- **P3 (agreed):** "3 visuals" = 3 distinct mental models / representations of the
  same concept, not 3 random pictures. This multi-representation idea is the thesis
  worth testing.
- **P4 (agreed):** With zero demand evidence, the immediate goal is a cheap
  validation test, not a polished platform.
- **P5 (agreed):** Correctness is the #1 technical risk. The architecture must make
  wrong visuals structurally hard, not just unlikely.

## Approaches Considered

### Approach A: Curated template library + AI router
Hand-build a library of tested parametric widgets (function grapher, vector field, free-body
diagram, unit circle, projectile sim, derivative-as-slope, reaction-energy, etc.).
The AI only classifies the input and parameterizes — it picks 3 templates and fills
in the formula/constants. It never generates rendering code.
- Effort: S–M · Risk: Low
- Pros: Correctness guaranteed (widgets are tested); instant render; cheap; no
  code-execution security risk; ships in days.
- Cons: Coverage capped at the library — "paste anything" breaks on the 16th concept
  type and must fall back. In tension with the founder's stay-broad instinct.
- Reuses: Recharts/D3/Plotly, mathjs, PhET-style interaction patterns.

### Approach B: AI generates the visual code on the fly
The LLM writes a self-contained interactive sketch (p5.js / canvas / SVG+JS) for the
specific concept, rendered in a sandboxed iframe. Truly paste-anything.
- Effort: M · Risk: High
- Pros: Maximum generality and demo wow; attempts literally anything.
- Cons: Correctness is unverifiable (violates P5); needs real sandboxing; higher
  latency/cost; and it's the closest thing to what ChatGPT already ships, so least
  differentiated. Worst possible failure mode for a trust tool.
- Reuses: sandboxed iframe runner, LLM code-gen, a viz prompt harness.

### Approach C: AI emits a structured spec, a deterministic renderer draws it
The LLM never emits code or pixels — it emits validated JSON against a schema the
product owns (e.g. `{type:"vector_field", equation, controls, annotations}`). A
hand-written deterministic renderer turns valid specs into interactive widgets. Before
anything renders, a **deterministic numeric/unit check is the binding correctness gate**:
evaluate the formula at sample points, check units/dimensions, assert spec-type invariants.
An optional LLM "does this representation fit the concept?" pass is **advisory only** — it
can flag, never certify. "3 visuals" = up to 3 distinct representation categories. Unknown
spec types degrade gracefully to a named fallback.
- Effort: M–L · Risk: Med
- Pros: Stays broad via an extensible schema (covers concept *families*, not single
  topics) AND structurally attacks correctness (deterministic renderer + validation).
  The schema + validator is a head-start, not a copyable prompt (the JSON schema itself
  is easy to copy; any real defensibility lives in validated coverage breadth over time —
  see Open Questions, treated as a bet to test, not an assumed moat).
- Cons: More upfront engineering than A; schema design is the hard intellectual work;
  still bounded by which spec types exist (but extensible over time).
- Reuses: JSON-schema validation, mathjs validation pass, renderer on canvas/SVG +
  charting libs, LLM structured output.

## Recommended Approach

**C as the engine, seeded by A.** The founder selected "C and A both," which is the
correct synthesis: A and C are not rivals — C is the architecture, A is how you seed
its vocabulary.

- **Engine = C:** structured JSON visualization spec schema + deterministic renderer
  + a math-validation pass that runs before render.
- **Seed content = A:** a canonical **~12 spec types** are hand-built, tested widget
  templates (the curated library). Every early visual is correct by construction. The
  validation prototype ships a **~6 spec-type subset** of those 12; the full ~12 is the
  near-term build target. Every count in this doc refers to this canonical pair (~12
  built / ~6 in the prototype).
- **AI's job:** classify the concept → select up to 3 spec types → parameterize each →
  validate the math → render. Never emits rendering code.
- **Representation-selection contract (resolves the "3 = 3 mental models" identity):**
  the 3 chosen spec types MUST span distinct representation *categories* (e.g.
  symbolic/graph, physical-analogy, procedural/step-through), not 3 variants of the
  same category. When a concept has fewer than 3 valid distinct representations, the
  product shows the 1–2 that are valid and says so — it never pads with a weak third or
  a duplicate. "3" is a ceiling and an aspiration, not a guarantee.
- **Broad input, coverage-limited v1 (resolves the breadth-vs-wedge tension):** v1
  accepts any STEM input (broad input surface) but its *correct* coverage is bounded by
  which spec types exist. So v1 is broad-by-aspiration, coverage-limited-by-build.
  Breadth is a discovery instrument — every fallback is a logged coverage gap that
  ranks the next spec type — not a v1 capability claim. Be honest in messaging about
  what is and isn't supported yet.
- **Graceful degradation (the "honest fallback," specified):** an input that maps to no
  known spec type shows an explicit message ("No interactive visual for this yet — here's
  a static explanation / here's what we do support") plus whatever partial render is
  valid, and logs the gap. It never shows a guessed or possibly-wrong widget. The exact
  copy is a product decision, but the behavior (named gap + no fabricated visual) is
  fixed. A B-style "generate something experimental" escape hatch is explicitly deferred,
  not in the first build.

Rationale: this is the only path that honors the founder's breadth decision while
directly attacking the correctness risk (P5) they agreed is #1, and it differentiates
structurally from ChatGPT's concept-triggered single modules.

## Open Questions

1. **ChatGPT competitive depth (HIGHEST PRIORITY).** OpenAI's interactive STEM
   visuals launched for 70+ concepts in 2026. Before building, the founder must
   directly use it on 10 real problems and document exactly where it fails:
   coverage gaps (the 71st concept), single-representation limitation, no
   problem-from-photo input, no persistence/study library. The wedge depends on
   these gaps being real and durable.
2. **Where does the wedge actually live** — concept-driven (top-of-funnel intuition)
   or problem-driven (the specific worksheet problem)? Validation should test both
   input modes and see which pulls repeat usage.
3. **Correctness guarantee boundary.** For which concept families can the validation
   pass actually certify correctness? Where it can't, does the product refuse rather
   than guess?
4. **Schema design.** What are the first ~12 spec types, and what concept families do
   they cover? This is the core engineering bet of Approach C.
5. **Who pays, eventually** — students directly, parents, or institutions? Not
   urgent pre-validation, but shapes which cohort's repeat usage matters most.

## Success Criteria

This is pre-product, so success = a clear validation signal, not shipped features.

- **Validation (next 2 weeks):** Watch 5+ real HS/college STEM students hit a
  genuinely confusing concept. Document what they reach for and the exact moment of
  confusion. At least 3 of 5 should describe a pain that the 3-representation idea
  plausibly addresses.
- **Prototype signal (if validation holds):** A narrow prototype (engine C + ~6 seed
  spec types) put in front of 20 target students. Success = a meaningful fraction
  return *unprompted* within a week and one subject/topic shows concentrated repeat
  usage (the emerging beachhead).
- **Kill signal:** Students try it once, say "cool," and never come back — or they
  say ChatGPT's built-in visuals already cover it. If that happens, the wedge is
  wrong and the doc should be revisited before more building.

## Distribution Plan

Web app with an existing deployment pipeline (e.g. Vercel) covers distribution — no
binary, package manager, or app store needed. No special CI/CD design required at
this stage beyond standard deploy-on-merge.

## Dependencies

- LLM API with reliable structured/JSON output (for the spec-generation step).
- A math evaluation/validation library (mathjs or equivalent) as the *foundation* for the
  correctness pass — note the library is glue; the per-spec-type correctness gate is not.
- A rendering stack for interactive widgets (canvas/SVG + a charting library).
- All component libraries are off-the-shelf, but there are **two pieces of genuinely novel
  engineering**, not one: (1) the spec schema design, and (2) the per-concept-family
  correctness-certification layer. Realistically (2) will need per-family custom validators
  rather than one generic numeric pass — Open Question 3 is unsolved and is real
  engineering risk, not library glue.

## The Assignment

**Before writing any code, do two things this week:**

1. **Use the competitor as a student would.** Open ChatGPT's interactive STEM visuals
   and Khan Academy on 10 real problems from an actual HS/college course. Write down,
   per problem, exactly where each one fails to give you intuition. This is your wedge
   evidence — or your kill signal.

2. **Watch 5 real students hit a wall.** Sit behind 5 HS/college STEM students (or
   recruit from a study Discord/subreddit) while they struggle with a concept. Do not
   help. Note what they reach for and the moment it stops clicking. Then show them a
   *paper sketch* of the same concept as 3 representations and ask which one they'd
   reach for first and why. You're testing the multi-representation thesis with zero
   code.

Report back what surprised you. A surprise is the signal that there's a real product
hiding here.
